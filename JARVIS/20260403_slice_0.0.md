# Slice 0.0 — Hardware Profile Provisioning and Backend Readiness

## Objective

Define and implement the repo structure that turns detected hardware facts into verified backend readiness before normal backend runtime selection occurs.

## Why this slice exists now

The repo can detect hardware capability and route toward backend/device choices, but it does not yet provision downstream hardware-specific dependencies into `backend/.venv` before backend runtime use. As a result, runtime can reach backend initialization with incomplete environment readiness and fall back late instead of using a verified path.

This slice establishes the missing structure so hardware detection leads to dependency provisioning, verification, and verified backend readiness.

Current repo evidence makes this gap explicit:

- hardware fact detection exists in `backend/app/hardware/detectors/`
- `backend/app/hardware/profiler.py` currently derives runtime recommendations directly from host facts
- `backend/app/runtimes/stt/local_runtime.py` and `backend/app/runtimes/tts/local_runtime.py` still discover CUDA/runtime problems late during normal runtime execution
- `scripts/ensure_models.py` owns model artifact presence only; it does not own backend-specific Python package provisioning or DLL/runtime verification
- the live acceptance path already exists and is evidence-proven through `backend/tests/runtime/test_profiler_live.py`, `backend/tests/runtime/test_slice1_stt_turn_live.py`, `backend/tests/runtime/test_slice2_tts_turn_live.py`, and `backend/tests/runtime/test_slice4_interruption_live.py`

This is therefore a corrective slice against the current repo state, not a greenfield design exercise. All implementation work under this slice must adjust existing code and preserve previously evidence-proven behavior unless a correction is explicitly required.

## Scope

This slice covers:

- hardware-profile manifest structure for additive dependency requirements
- one provisioning/preflight owner for hardware-profile dependency readiness
- mapping detector facts to hardware-profile requirements
- backend-readiness verification for STT first
- capability/profiler integration with verified backend readiness
- setup/bootstrap/validation alignment so readiness is checked before ordinary runtime use
- the general pattern for later TTS and LLM backend readiness
- holistic correction of existing profiler, runtime, bootstrap, and acceptance surfaces so proven work is realigned instead of bypassed

This slice uses CUDA/STT as the first concrete implementation track, but the structure must support all relevant hardware profile dimensions:

- CPU
- GPU without CUDA
- GPU with CUDA
- NPU
- combined hardware cases
- OS differences
- memory profile distinctions

## Out of scope

This slice does not complete:

- TTS backend CUDA readiness implementation
- LLM backend CUDA or accelerator readiness implementation
- broad redesign of unrelated runtime orchestration
- replacement of the existing model catalog / model acquisition flow in `config/models/*.yaml`, `backend/app/models/catalog.py`, `backend/app/models/manager.py`, and `scripts/ensure_models.py`
- live feature work unrelated to provisioning/readiness
- ad hoc runtime self-repair during normal backend execution
- a one-off CUDA-only patch that does not generalize to other hardware profile dimensions

## Target structure

The repo should gain a deterministic flow:

1. detectors produce hardware facts
2. hardware facts resolve into one or more hardware-profile manifests
3. provisioning/preflight reads the applicable manifests
4. additive dependencies are ensured in `backend/.venv`
5. required package/DLL/runtime evidence is verified
6. backend readiness is recorded
7. profiler/capability routing consumes verified backend readiness
8. runtime selection uses verified readiness rather than host capability alone

The corrected repo flow should align existing surfaces instead of replacing them:

- fact production remains in `backend/app/hardware/detectors/*.py`
- report assembly remains profiler-first in `backend/app/hardware/profiler.py`
- runtime selectors remain thin execution selectors in `backend/app/runtimes/stt/stt_runtime.py`, `backend/app/runtimes/tts/tts_runtime.py`, and later LLM selection surfaces
- model artifact ownership remains in the existing catalog/manager/ensure-models path
- validation remains anchored in existing unit/runtime suites and `scripts/validate_backend.py`

The configuration shape should support:

- base dependency authority:
  - `backend/requirements.txt`

- additive hardware/backend manifests:
  - `config/hardware/*.json`

- verified backend-readiness state:
  - a readiness structure emitted alongside current profile/capability outputs, likely adjacent to `CapabilityFlags` inside `backend/app/core/capabilities.py`

Each manifest should be able to represent:

- profile identity
- applicability conditions
- backend scopes
- additive package requirements
- required verification evidence
- optional OS and memory qualifiers
- merge/additive behavior when multiple manifests apply

## Key design decisions

1. **Detectors produce facts, not provisioning actions**
   - `backend/app/hardware/detectors/cpu_detector.py`
   - `backend/app/hardware/detectors/cuda_detector.py`
   - `backend/app/hardware/detectors/gpu_detector.py`
   - `backend/app/hardware/detectors/memory_detector.py`
   - `backend/app/hardware/detectors/npu_detector.py`
   - `backend/app/hardware/detectors/os_detector.py`
   should remain fact producers.

2. **Provisioning ownership is centralized**
   One provisioning/preflight owner should interpret resolved hardware-profile requirements, install missing additive dependencies into `backend/.venv`, verify readiness, and expose the result to profiler/runtime selection. This owner should complement existing model-artifact ownership rather than duplicate it.

3. **Backend readiness is separate from host capability**
   Host CUDA presence alone does not imply backend CUDA usability. Current `profile.cuda_available` and current recommendation fields are necessary inputs, but they are not sufficient proof of backend readiness.

4. **Backend readiness must be backend-scoped and device-scoped**
   The structure should support separate verified readiness states such as:
   - STT CPU ready
   - STT CUDA ready
   - TTS CPU ready
   - TTS CUDA ready
   - LLM local ready
   - future NPU/vendor-specific ready states

5. **`backend/.venv` is the only environment**
   All provisioning, install, verification, and readiness probes must target `backend/.venv`.

6. **`backend/requirements.txt` remains base authority**
   Hardware-profile manifests contribute additive requirements; they do not replace the base requirements file.

7. **Runtime consumes verified state and fails closed**
   Normal backend runtime selection should follow verified readiness. Runtime code may retain defensive guards, but it should not be the primary place that missing dependencies are first discovered.

8. **Existing evidence-proven paths must be corrected, not bypassed**
   This slice must adjust the current profiler-first and live-runtime-proven architecture rather than introducing a parallel bootstrap path, parallel selector path, or shadow acceptance harness.

9. **Model artifacts and Python dependencies remain separate concerns**
   `scripts/ensure_models.py`, `backend/app/models/catalog.py`, and `backend/app/models/manager.py` currently govern model artifact presence. Hardware-profile provisioning should govern additive Python package/runtime readiness. These flows may coordinate, but they should not be conflated.

10. **Combined hardware cases should compose additively**
    The structure should not require one monolithic manifest per permutation. A Windows + NVIDIA CUDA + high-memory + STT manifest combination is preferable to exploding every permutation into a single profile file.

## Sub-slices

The sub-slices below are intentionally thin and corrective. Each one must adjust the current repo state and preserve prior evidence-proven behavior while moving readiness discovery earlier and making it explicit.

## Sub-Slice 0.0.1 — Hardware profile manifest structure

Define the configuration structure for hardware-profile manifests that express additive dependency and verification requirements.

### Goal

Introduce a repo-real manifest structure that can describe:

- CPU profiles
- GPU non-CUDA profiles
- GPU CUDA profiles
- NPU profiles
- combined hardware profiles
- OS and memory qualifiers
- backend-scoped additive requirements

### Expected repo changes

- add a hardware-profile manifest location, likely under:
  - `config/hardware/`
- define initial manifest files for the first implementation track and for the baseline categories the resolver must understand
- define manifest fields for:
  - applicability
  - backend scope
  - additive packages
  - required evidence
  - merge precedence / additive behavior
- keep manifests aligned to existing runtime/model labels already in use in:
  - `config/models/stt.yaml`
  - `config/models/tts.yaml`

### Validation

- manifest files parse cleanly
- manifest structure is sufficient to represent the first STT CUDA track on the current repo state
- manifest structure can represent GPU-non-CUDA, Intel/Qualcomm NPU, and memory-qualified profiles without redesign
- manifest structure can represent later TTS and LLM readiness without redesign

### Notes

Use CUDA/STT as the first concrete example, but the manifest structure must remain general enough for all hardware profile dimensions. Do not duplicate the existing full model catalog inside these manifests; reference existing runtime/model identities where possible.

## Sub-Slice 0.0.2 — Hardware fact to profile resolution

Define how detector facts resolve into one or more applicable hardware-profile manifests.

### Goal

Establish a deterministic mapping from detector outputs to hardware-profile requirements without turning detectors into provisioning code.

### Expected repo changes

- add a hardware-profile resolver layer adjacent to the existing hardware-profiler flow
- resolve against current fact fields already emitted in `HardwareProfile`, especially:
  - `os`
  - `arch`
  - `gpu_available`
  - `gpu_vendor`
  - `cuda_available`
  - `npu_available`
  - `npu_vendor`
  - `memory_total_gb`
  - `device_class`
- support composable profile resolution where multiple manifests may apply together
- avoid overloading current `profile_id` into the sole dependency/provisioning authority

### Validation

- CPU host resolves correctly
- GPU without CUDA resolves correctly
- GPU with CUDA resolves correctly
- NPU resolves correctly
- combined hardware facts can resolve to multiple additive manifests where appropriate

### Notes

The resolver should support additive merge behavior rather than forcing every hardware permutation into one giant mutually-exclusive profile. It should correct the current gap between host facts and downstream runtime readiness without breaking profiler-first authority.

## Sub-Slice 0.0.3 — Provisioning/preflight owner

Add the single provisioning/preflight owner that ensures additive dependencies are present in `backend/.venv` before runtime use.

### Goal

Create the owner that:

- reads resolved manifests
- checks additive package presence
- installs missing additive packages into `backend/.venv`
- verifies required evidence
- records backend readiness

### Expected repo changes

- add a provisioning/preflight surface under the hardware/readiness area and/or an explicit script surface for controlled invocation
- add package presence checks against `backend/.venv`
- add controlled installation behavior into `backend/.venv`
- add package/DLL/runtime verification behavior
- coordinate with existing `scripts/ensure_models.py` and model-artifact presence instead of replacing it

### Validation

- missing additive packages are detected deterministically
- required additive packages are installed into `backend/.venv`
- verification evidence is checked after install
- provisioning result is stable and repeatable
- ordinary backend runtime no longer serves as the first dependency discovery surface

### Notes

This owner is the missing rail between hardware detection and backend runtime selection. It should be an explicit preflight/bootstrap surface, not hidden runtime self-repair during ordinary backend execution. Existing runtime fallbacks should remain only as defensive guards while the new readiness path is established.

## Sub-Slice 0.0.4 — STT CUDA readiness track

Implement the first real backend-readiness track using STT CUDA as the baseline case.

### Goal

Provision and verify STT CUDA readiness end-to-end using the new manifest + provisioning structure.

### Expected repo changes

- define STT CUDA additive dependency requirements in hardware-profile manifests
- define required STT CUDA evidence, including package and DLL/runtime evidence
- add an STT-specific readiness verification/probe separate from the live voice turn itself
- adjust current STT selection/runtime behavior so verified readiness, not late fallback discovery, drives device selection
- preserve the existing CPU-ready degraded path as an explicit alternate ready path rather than an accidental late fallback

### Validation

- STT CUDA dependency requirements are correctly recognized
- required STT CUDA evidence is verified before normal runtime selection
- STT backend readiness is only marked usable after verification
- when CUDA readiness is not verified, a verified CPU-ready STT path is selected before `run_voice_turn(...)`
- the live STT acceptance path remains clean

### Notes

This is the first concrete track and the first real acceptance gate for the new provisioning/readiness structure. The current late CUDA fallback behavior in `backend/app/runtimes/stt/local_runtime.py` should become a defensive last guard, not the normal path that discovers the environment is incomplete.

## Sub-Slice 0.0.5 — Capability/profiler integration

Update capability/profiler generation so it consumes verified backend readiness rather than host capability alone.

### Goal

Make profiler/capability routing reflect verified backend readiness.

### Expected repo changes

- add/extend capability-report structures to carry backend readiness explicitly
- prefer a readiness structure adjacent to `CapabilityFlags` rather than overloading host-capability booleans with verified runtime state
- update `backend/app/hardware/profiler.py` so recommendations are readiness-backed
- update `backend/app/runtimes/stt/stt_runtime.py` and `backend/app/runtimes/tts/tts_runtime.py` so selectors consume verified readiness-backed recommendation data
- preserve the current profiler-first authority model while correcting what that authority actually means

### Validation

- capability generation distinguishes host capability from backend readiness
- STT device recommendation follows verified STT readiness
- safe degradation remains explicit when readiness is not verified
- current report consumers can be migrated without breaking previously proven behavior

### Notes

Profiler/capability integration should consume proven readiness, not invent policy from host facts alone. This is a correction to the current authority contract, not a replacement of the profiler-first architecture.

## Sub-Slice 0.0.6 — Bootstrap, validation, and acceptance alignment

Align setup, bootstrap, and validation surfaces so readiness is explicit and testable before normal runtime use.

### Goal

Make hardware/backend readiness part of the intentional setup and validation path.

### Expected repo changes

- add an explicit readiness invocation path for operators/bootstrap flows
- keep model-ensure ownership and hardware-readiness ownership distinct, even if a higher-level bootstrap command orchestrates both
- extend targeted unit coverage for manifest resolution and readiness logic
- extend runtime validation to prove the first concrete readiness-backed STT path
- align `scripts/validate_backend.py` and existing live/runtime acceptance surfaces with the new readiness boundary

### Validation

- there is one deterministic operator/bootstrap path to prove readiness before normal use
- unit tests cover manifest parsing/resolution/readiness logic
- existing live tests remain the real acceptance surfaces for the corrected flow
- regression coverage remains intact for profiler, STT, TTS, and interruption behavior

### Notes

Do not introduce a shadow harness. Existing acceptance surfaces such as `backend/tests/runtime/test_profiler_live.py`, `backend/tests/runtime/test_slice1_stt_turn_live.py`, `backend/tests/runtime/test_slice2_tts_turn_live.py`, and `backend/tests/runtime/test_slice4_interruption_live.py` are already the meaningful runtime constraints and should stay authoritative.

## Sub-Slice 0.0.7 — Extension pattern for TTS and LLM

Define the follow-up readiness pattern for TTS and LLM without implementing those backend tracks fully in this slice.

### Goal

Lock in the shared pattern so later TTS and LLM readiness slices follow the same structure.

### Expected repo changes

- ensure the shared readiness schema supports backend-specific readiness records beyond STT
- ensure manifests can express TTS- and LLM-specific additive packages and verification evidence later
- prepare selection surfaces so future TTS/LLM readiness tracks can be integrated without redesign

### Validation

- TTS and LLM have clear follow-up implementation paths
- no STT-specific assumptions are embedded into the shared structure

### Notes

Do not assume TTS or LLM use the same dependency/runtime behavior as STT. TTS may require different package and accelerator evidence. LLM may require local package readiness, service reachability, or provider-specific verification depending on the chosen runtime path.

## Risks / sequencing notes

- base dependency authority and additive manifest requirements must remain consistent
- provisioning must remain scoped to `backend/.venv`
- the current `backend/requirements.txt` already contains CUDA-adjacent packages; any transition to additive manifests must preserve working base behavior while the authority split is corrected
- detector surfaces should not be turned into provisioning code
- capability generation must not overclaim backend usability before readiness verification
- `scripts/ensure_models.py` is already a proven model-artifact path and should not be destabilized while solving Python dependency readiness
- current late fallback logic in STT/TTS runtimes is evidence-proven behavior and must be demoted carefully rather than removed blindly
- TTS and LLM must not inherit STT-specific assumptions
- combined hardware cases should resolve via additive manifests without forcing unnecessary profile explosion
- OS-specific runtime evidence must be represented honestly, especially Windows DLL evidence vs Linux/macOS shared-library/runtime evidence
- validation must prioritize real runtime acceptance over partial import/probe success
- each sub-slice must adjust existing code and previously evidence-proven work holistically; no shadow workflow should be introduced to avoid touching the real repo path

The correction is only done when the repo reaches that state by adjusting the existing codebase holistically and preserving the already proven live/runtime path as the authority.
