# 20260329-slice_1.6.md
## Sub-Slice 1.6: Model Acquisition — Ensure-Present Mechanism

**Date:** 2026-03-29
**Amends:** `20260329-slice_1.md` — inserted before Sub-slice 1.5
**Status:** Required before Slice 1.5 live validation can proceed

**Problem being solved:**
`faster-whisper` will auto-download Whisper models into the HuggingFace cache
on first use if no explicit model path is given. This produces:
- model artifacts at an uncontrolled, implicit location
- no reproducible verification path
- no foundation for future llama.cpp model acquisition (which has no auto-download)
- no integration between `config/models/stt.yaml` and what is actually on disk

Sub-slice 1.6 introduces a structured, config-driven model acquisition mechanism
that resolves all four problems and extends cleanly to TTS and LLM models.

---

## Design Principles

1. **`config/models/` is the catalog — not code.** The acquisition mechanism reads
   model definitions from existing YAML files. It does not maintain a parallel manifest.

2. **`models/` is the artifact directory.** Per `repo_tree.md`, `models/` stores
   downloaded runtime assets. Contents are `.gitignore`d. Structure is tracked.

3. **Three operations, one interface:** `verify`, `download`, `ensure`.
   - `verify` — check presence and integrity of a model artifact; return status
   - `download` — fetch the model to `models/<family>/<model_name>/`; resume-capable
   - `ensure` — verify first; download only if absent (primary runtime operation)

4. **Config drives the source.** Each model entry in `config/models/stt.yaml` gains
   a `hf_repo_id` field. The acquisition module reads this field to resolve the
   download source. No source URL hardcoded in Python.

5. **Fail closed.** If a model is required and neither present nor downloadable,
   the caller receives an explicit `ModelNotAvailableError` — never a silent fallback
   to an uncontrolled cache location.

6. **Callable as both a backend module and a CLI script.**
   - `backend/app/models/manager.py` — importable by `voice_service.py` at startup
   - `scripts/ensure_models.py` — standalone invocation for setup and CI

**Extension pattern for future catalogs:**
When `config/models/llm.yaml` gains entries (GGUF files, llama.cpp models), they follow
the same schema — `hf_repo_id` and `local_dir` per entry. The acquisition module
is called identically regardless of model family.

## Files to Create

**`backend/app/models/__init__.py`** — empty package marker

**`backend/app/models/manager.py`**

Public interface:
```python
class ModelNotAvailableError(Exception): ...

def verify_model(local_dir: str) -> bool:
    """Return True if local_dir exists and is non-empty. No network calls."""

def download_model(hf_repo_id: str, local_dir: str) -> str:
    """Download model from HuggingFace to local_dir. Resume-capable.
    Returns local_dir on success.
    Raises ModelNotAvailableError on download failure."""

def ensure_model(hf_repo_id: str, local_dir: str) -> str:
    """Verify first. Download only if absent.
    Returns local_dir if present or after successful download.
    Raises ModelNotAvailableError if model cannot be made available."""
```

Implementation notes:
- `verify_model`: `Path(local_dir).exists() and any(Path(local_dir).iterdir())`
- `download_model`: uses `huggingface_hub.snapshot_download(repo_id, local_dir=local_dir)`
  with `local_files_only=False`; wraps all HF exceptions in `ModelNotAvailableError`
  with a message that names both the `hf_repo_id` and `local_dir` attempted
- `ensure_model`: calls `verify_model` first; calls `download_model` only if `False`
- All three functions are safe to call multiple times (idempotent)
- No global state; no singletons; pure functions

**`backend/app/models/catalog.py`**

```python
def load_stt_catalog(config_path: str = "config/models/stt.yaml") -> dict:
    """Load and return the full stt.yaml model catalog as a dict."""

def get_model_entry(model_name: str,
                    config_path: str = "config/models/stt.yaml") -> dict:
    """Return the catalog entry for a named STT model.
    Raises KeyError with named model if not found in catalog."""
```

This is the bridge between `config/models/stt.yaml` and the manager.
Callers use `get_model_entry(flags.stt_recommended_model)` to resolve
`hf_repo_id` and `local_dir` without hardcoding either in application code.

**`scripts/ensure_models.py`**

Standalone script, callable from PowerShell:
```
backend/.venv/Scripts/python scripts/ensure_models.py --family stt --model whisper-large-v3-turbo
backend/.venv/Scripts/python scripts/ensure_models.py --family stt --all
backend/.venv/Scripts/python scripts/ensure_models.py --verify-only --family stt --model whisper-large-v3-turbo
```

Behavior:
- `--family stt` reads `config/models/stt.yaml`
- `--model <name>` ensures a single named model
- `--all` ensures all models in the catalog
- `--verify-only` runs `verify_model` only; prints PRESENT / MISSING per model; exits non-zero if any are missing
- prints explicit progress per model: `[DOWNLOAD] whisper-large-v3-turbo → models/stt/whisper-large-v3-turbo`
- prints `[PRESENT] whisper-large-v3-turbo` if already on disk
- exits non-zero on any `ModelNotAvailableError`

## `FasterWhisperSTT` Integration

After 1.6 is implemented, `FasterWhisperSTT.transcribe()` in
`backend/app/runtimes/stt/local_runtime.py` must be updated to:

1. Resolve `local_dir` from `catalog.get_model_entry(self.model_name)`
2. Call `ensure_model(hf_repo_id, local_dir)` before loading the model
3. Pass `local_dir` as the model path to `WhisperModel(local_dir, ...)`
   instead of passing the raw model name (which triggers HF auto-download to cache)

This is the only change to `local_runtime.py`. The model path becomes explicit
and controlled rather than implicit and cache-dependent.

## `.gitignore` Requirement

`models/` directory contents must be `.gitignore`d.
If `.gitignore` does not yet exist or does not cover `models/`, it must be updated:

```
# Downloaded model artifacts — do not commit
models/stt/
models/tts/
models/llm/
```

The `models/` directory itself and any `.gitkeep` markers within it are tracked.
The downloaded model files are not.
